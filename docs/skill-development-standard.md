# OpenClaw 技能开发规范 V1.0
（基于《The Complete Guide to Building Skills for Claude》最佳实践）

## 一、技能定义
技能是一组封装好的指令、工具和工作流，教会AI如何处理特定任务或流程，实现"一次配置、多次复用"，避免每次重复解释偏好、流程和领域知识。

## 二、技能分类
### 分类1：独立技能（Standalone Skills）
- 不需要调用外部工具，纯指令驱动
- 示例：文档写作规范、项目管理流程、PPT生成模板
- 特点：零依赖、易复用、适合标准化流程

### 分类2：工具增强技能（Tool-Enhanced Skills）
- 依赖外部工具/API完成任务
- 示例：Git工作流、天气查询、文件处理
- 特点：功能强大、需要工具权限、适合复杂操作

### 分类3：MCP集成技能（MCP-Integrated Skills）
- 深度集成模型上下文协议，支持多Agent协同
- 示例：多Agent项目管理、OIAFlow协同工作流
- 特点：高度自动化、跨系统协同、适合复杂业务流程

## 三、技能结构标准（强制执行）
```
skill-id/
├── skill.md            # 技能元数据 + AI调用说明（给AI看）
├── README.md           # 用户使用文档（给人看）
├── manifest.json       # 技能包元数据（技能市场安装用）
├── package.json        # 依赖配置（有代码的技能必填）
├── prompts/            # 提示词模板目录（可选）
│   ├── system.md       # 系统提示词
│   └── user.md         # 用户提示词模板
├── scripts/            # 可执行脚本目录（可选）
│   └── main.mjs        # 主执行脚本
├── templates/          # 输出模板目录（可选）
│   └── report.md       # 报告/输出模板
├── tests/              # 测试用例目录（必填）
│   └── test-cases.json # 测试案例和验收标准
└── assets/             # 资源文件目录（可选）
    └── images/         # 图片等资源
```

## 五、开发最佳实践
### 1. 设计原则
- **单一职责**：每个技能只做好一件事，避免大而全
- **可组合性**：技能之间可以互相调用，形成更复杂的工作流
- **配置驱动**：可变参数放在配置文件，避免硬编码
- **容错设计**：包含错误处理、重试机制、用户友好的错误提示
- **结果可验证**：每个技能的输出都要有明确的验收标准

### 2. 命名规范
- 技能ID：小写英文，连字符分隔，如`git-workflow`
- 版本号：语义化版本，`主版本号.次版本号.修订号`
  - 主版本号：不兼容的API变更
  - 次版本号：新增功能，向后兼容
  - 修订号：bug修复，向后兼容

### 3. 安全规范
- 最小权限原则：只申请必要的权限
- 敏感信息保护：不硬编码API密钥、密码等敏感信息
- 输入校验：所有用户输入都要做校验，防止注入攻击
- 输出审查：确保输出不包含敏感信息、不实信息

## 六、测试流程（强制执行）
每个技能发布前必须通过以下测试：

### 1. 单元测试
- 测试基本功能是否正常工作
- 测试所有参数组合
- 测试正常输入的输出是否符合预期

### 2. 边缘案例测试
- 测试异常输入的处理
- 测试边界值情况
- 测试网络/服务不可用的降级处理

### 3. 集成测试
- 测试与其他技能的协同工作
- 测试与外部工具/API的集成
- 测试在完整工作流中的表现

### 4. 用户验收测试
- 真实场景下的使用验证
- 确认用户体验流畅
- 确认输出符合用户预期

## 七、发布流程
1. 完成所有测试
2. 更新版本号
3. 更新更新日志
4. 生成发布包
5. 上传到技能市场
6. 发布版本说明
